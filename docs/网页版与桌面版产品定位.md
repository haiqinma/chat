# 网页版与桌面版产品定位

本文档用于明确 Chat 网页版和桌面版的产品定位、当前能力边界和后续功能评估原则。

当前阶段先把两个版本规划清楚，不提前引入复杂的多运行时、组织权限或市场后端设计。后续增加 Skill、Tool、本地文件、命令执行等能力时，应先根据本文档判断应该支持哪个版本，再决定具体实现。

## 1. 核心结论

- 网页版是当前主要产品形态，重点是通过 Router 提供开箱即用的在线 AI 能力。
- 桌面版复用网页版的账户、Router、会话和 Skill 体系，并逐步增加操作系统集成能力。
- 桌面版不是另一套产品，也不单独建设一套 Skill 市场。
- 网页版不能访问用户电脑上的文件或启动用户本机进程。
- standalone 服务端能够运行的 Tool 属于部署实例能力，由实例的运行配置决定，不等于网页用户的本地能力。
- 桌面版是否支持本地 Tool Runtime，后续根据真实需求、安全边界和维护成本单独评估，当前不作为默认承诺。

## 2. 网页版定位

网页版定位为：

> 登录后即可使用的在线 AI 工作台。

当前主要部署方式是 Next.js standalone。用户完成登录并获得 Router 模型后，应当可以直接使用通用问答、AI 绘画、实时聊天和不依赖本地环境的 Skill。

网页版重点保证：

- 登录、账户切换和数据同步稳定。
- Router 模型开箱即用。
- 会话、Skill 配置和用户偏好按账户隔离。
- Skill 的模型能力要求能够根据 Router 模型 tags 判断。
- 云端或部署环境已经提供的 Tool 可以被指定 Skill 使用。
- 不支持的能力明确说明原因，不提供点击后没有结果的入口。

网页版不应直接提供：

- 用户本机文件系统访问。
- 用户本机 Git、Shell 或本地命令执行。
- 从浏览器启动本地 MCP/Tool Server。
- 网页账户在 standalone 服务器上配置任意 command 和 args。

### 2.1 standalone Tool 的含义

当前 standalone Tool Runtime 运行在 Next.js 服务端进程中。

当前产品没有管理员角色或管理后台。Tool 是否可用，由 standalone 实例启动时的环境变量和配置文件决定。

其他架构文档中出现的“管理员”“组织权限”等说法，属于未来市场和组织治理的规划，不代表当前 Chat 已经提供对应的角色、后台或权限系统。

因此：

- Tool 实际运行在 Chat 部署服务器，而不是访问者的电脑。
- `ENABLE_TOOLS` 和 `tool_config.json` 属于部署环境配置。
- 公共或多人部署中，网页账户不能通过产品界面修改实例级 Tool 配置。
- 网页账户只能使用当前部署实例已经提供的 Tool。
- 用户自己的 API Key 等私有配置如果需要支持，必须按账户隔离，不能继续作为实例级公共配置处理。

单用户自托管场景中，部署者和使用者可能是同一个人。部署者可以在 Chat 产品之外修改环境变量和配置文件，但产品概念仍应区分“实例提供 Tool”和“账户使用 Tool”。

## 3. 桌面版定位

桌面版定位为：

> 在在线 AI 工作台基础上，提供更好的桌面交付和操作系统集成。

桌面版继续复用：

- 同一套账户和登录体系。
- 同一套 Router 模型。
- 同一套会话和数据同步。
- 同一套 Skill Package 和发现市场。
- 与网页版一致的主要交互和业务逻辑。

当前桌面版采用 Tauri 静态导出。现阶段优先保证：

- 桌面开发、构建、签名和更新链路稳定。
- 登录、Router、会话、Skill 等核心功能与网页版一致。
- 剪贴板、文件保存、通知和应用更新等桌面能力稳定。
- 桌面专用改动不反向影响 Web 主链。

当前桌面版的 Tool Runtime 是禁用状态。后续是否增加本地 Tool Runtime，需要单独评估：

- 用户是否存在明确的本地文件、Git 或命令执行需求。
- Tauri 权限提示和授权模型是否清晰。
- 本地配置和密钥能否安全保存。
- 是否具备命令白名单、目录授权和操作确认机制。
- macOS、Windows 和 Linux 的维护成本是否可接受。

在这些问题明确前，不在桌面发现页中展示无法真正使用的本地 Tool 操作。

## 4. 两个版本共享与差异

| 能力                      | 网页版            | 桌面版         |
| ------------------------- | ----------------- | -------------- |
| 账户、登录和数据同步      | 支持              | 支持           |
| Router 模型               | 支持              | 支持           |
| 通用问答                  | 支持              | 支持           |
| AI 绘画                   | 支持              | 支持           |
| 实时聊天                  | 支持              | 支持           |
| Skill 市场与配置          | 支持              | 支持           |
| 部署实例提供的服务端 Tool | standalone 可支持 | 当前不直接复用 |
| 云端/远程 Tool            | 可以逐步支持      | 可以逐步支持   |
| 用户本机文件、Git、Shell  | 不支持            | 后续评估       |
| 本地 Tool Runtime         | 不支持            | 后续评估       |
| 桌面通知、保存、更新      | 浏览器能力范围内  | 原生支持       |

## 5. Skill 和 Tool 的简单规则

当前阶段只保留以下规则。

### 5.1 Skill

- Skill 是用户使用 AI 能力的主要入口。
- 同一个 Skill Package 同时服务网页版和桌面版。
- Skill 声明需要的模型能力和 Tool。
- 当前版本缺少必要能力时，Skill 应显示不可用原因。
- 不为网页版和桌面版复制两套同名 Skill。

“启用 Skill”表示：

- 将 Skill 添加到当前账户。
- 保存当前用户的模型和运行配置。
- 允许它出现在新建会话和“我的 Skill”中。

启用 Skill 不表示后台部署服务。

### 5.2 Tool

Tool 是 Skill 依赖的执行能力，不是通用会话默认能力。

网页版中的 Tool 分两种情况：

- 当前部署实例已经提供：用户可以在允许范围内使用。
- 当前部署没有提供：用户看到不可用原因，不执行隐式安装或降级。

桌面版当前不承诺本地 Tool。后续如果实现，本地 Tool 的安装、权限和状态只属于当前设备，不能简单等同于账户已经启用。

### 5.3 会话

- 通用问答不携带 Tool。
- 只有明确绑定 Tool 的 Skill 才能发送和执行工具调用。
- 会话使用创建时的 Skill 配置，不能因为全局 Tool 状态变化而自动获得额外能力。

## 6. 发现页的当前职责

发现页当前只负责：

- 展示 Skill 和 Tool。
- 说明当前版本是否可用。
- 配置和启用当前环境真正支持的能力。
- 将已经启用的 Skill 交给用户使用。

发现页不负责：

- 为普通网页用户部署服务端进程。
- 在网页中安装用户本机 Tool。
- 自动把所有 Tool 注入通用会话。
- 用一个“启用”按钮同时表达部署、安装、授权和使用。

当前阶段按钮含义应尽量直接：

- Skill：`配置并启用`、`使用`、`配置`。
- 已部署的网页 Tool：`配置`、`启用`、`测试`。
- 网页环境不支持的本地 Tool：`需要桌面版` 或 `当前版本不可用`。
- 桌面版尚未实现的本地 Tool：不展示可执行的安装按钮。

## 7. 后续功能评估清单

后续每次增加能力前，先回答以下问题：

1. 这个能力属于网页版、桌面版，还是两个版本都需要？
2. 它运行在 Chat 服务器、远程服务，还是用户电脑？
3. 配置发生在部署实例，还是当前账户或设备？
4. 配置是账户级数据，还是当前设备的本地数据？
5. 当前版本不支持时，页面如何明确提示？
6. 是否会让通用问答意外获得 Skill 专属能力？
7. 是否会影响另一个版本的构建或运行链路？

这些问题没有明确答案时，先不进入实现。

## 8. 当前迭代重点

### 网页版

- 保持 Router 开箱即用。
- 完善登录、账户切换和数据同步。
- 明确 Skill 的配置、启用和使用流程。
- 明确服务端 Tool 是部署实例能力，不通过用户界面动态部署。
- 让发现页只展示当前环境真实可用的操作。

### 桌面版

- 保持核心功能与网页版一致。
- 保证 Tauri 构建、签名、更新和桌面基础能力稳定。
- 暂不把 standalone 的服务端 Tool 配置直接搬到桌面版。
- 根据真实使用需求再评估本地文件、Git、Shell 和本地 Tool Runtime。

## 9. 暂不处理的内容

本文档当前不展开：

- 完整的 Tool 多运行时抽象。
- 企业组织级部署和授权。
- Tool 计费、审计和配额。
- 市场后端和付费能力。
- 本地 Agent 和跨设备调度。
- 桌面端本地模型。

这些能力后续有明确需求时，再单独设计。
